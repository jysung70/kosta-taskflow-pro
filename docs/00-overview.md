## 읽는 순서

```
00-overview  →  01-product  →  02-specs  →  03-design  →  04-tasks  →  05-conventions
   (전체 맥락)      (무엇을)       (어떻게)      (어떻게 보이나)   (언제까지)     (어떻게 짜나)
```

문서를 이 순서로 읽으면 **"왜 만드는가 → 무엇을 만드는가 → 어떻게 만드는가"** 의 흐름이 자연스럽게 연결된다.

---

## 왜 이렇게 나누는가 — 관심사 분리 원칙

소프트웨어 설계에서 **관심사 분리(Separation of Concerns)** 는 하나의 단위가 하나의 책임만 갖도록 나누는 원칙이다. 이 문서 구조는 동일한 원칙을 문서에 적용한 것이다.

| 혼합했을 때 문제 | 분리했을 때 이점 |
|---|---|
| 기획이 바뀌면 기술 명세 문서 전체를 수정해야 한다 | `01-product.md` 만 수정하면 된다 |
| 디자인 변경이 작업 일정과 섞여 파악이 어렵다 | `03-design.md` 와 `04-tasks.md` 가 독립적으로 관리된다 |
| 코드 규칙을 찾으려면 문서 전체를 뒤져야 한다 | `05-conventions.md` 한 곳만 보면 된다 |
| 신규 팀원이 전체 문서를 처음부터 읽어야 한다 | 필요한 관심사의 파일만 선택해서 읽을 수 있다 |

각 파일은 **변경 이유가 하나** 다. 제품 방향이 바뀌면 `01-product.md`, 기술 스택이 바뀌면 `02-specs.md`, 컨벤션이 바뀌면 `05-conventions.md` 만 열면 된다.
